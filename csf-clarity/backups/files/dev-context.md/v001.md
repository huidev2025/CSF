### 中段执行

#### 三问承接（每次取到新 STB 时）

取到 STB 后**不立即动键盘**。在 devlog 中按以下模板填写，填完汇报 Owner，确认后再开始编码。

> ⚠️ **STB 读起来越简单直接，越是跳过三问的高风险时刻。** 无论 STB 复杂度如何，模板必须填写。

```
【三问承接 STB: xxx】
Q0-沉淀标签核查:
  DT-1: [🔵TASK / 🟢PACK / 🟣FLDD 已标] / [⚠️ 缺标 → PENDING]
  DT-2: [...]（逐条列）

Q1-大图: [产品中的位置] / [本次 IT 在 TP 的第几个 IT]

Q2-文件范围:
  改: [文件路径]
  新建: [文件路径]
  参考: [文件路径]
  疑问: [有/无，有则列出]

Q3-验收对齐:
  DT-1 → 验收档位: [🟢强 / 🔵中 / 🟡弱] → 证明方式: [具体说明]
  DT-2 → [...]

状态: ✅ 全部确认 → 汇报 Owner 后开始编码
      🟡 有疑问/缺标 → assign 参谋长 @MSG:N → 等回馈后再动手
```

**时序约束**：devlog 中本模板的写入，必须早于该 STB 对应的任何代码文件改动。不得写完代码后补填三问。

#### 编码纪律（FLDD 全程适用）

**① 三栏自检**（承接 STB 时通过三问 Q0/Q2/Q3 体现）：

| 栏 | 含义 |
|---|---|
| ✅ 已读懂 | STB 里我能直接动手的部分 |
| 🟡 不确定 | 我读出歧义 / 多解 / 缺料的点 |
| 🔵 准备假设 | 我打算按 X 假设处理，请校验 |

任一 🟡/🔵 非空 → **不动键盘，发 PENDING 回参谋长**。三栏仅 ✅ 非空才进入编码。

**② 范围契约** 🟣FLDD：每一行修改必须能直接追溯到 STB 的某个 DT。范围外动代码（含调格式、改 import、修 typo）= 立 PENDING，不直接改。

**③ 等价检验** 🟣FLDD：新代码体量 > 旧等价代码 2 倍 → 停下在 devlog 写明原因，参谋长校验后继续。

**④ 发现不一致** 🟣FLDD：编码中发现"实际情况与 STB 描述矛盾"→ 停下 PENDING 报参谋长，不自行假设。

**⑤ 写前必读**（守则§3）🟣全局：写任何文件前，依据的事实必须来自本会话最近一次真读。中间有其他操作间隔 → 重读后再写。

**⑥ 凭印象触发词**：以下表述出现即为凭印象信号，必须停下重读对应文件后再继续：
- "我记得……" / "上次好像是……" / "按印象应该……"
- "文件里有X"（但未在本会话最近读过该文件）

#### 轻量变更（无 STB 的日常改动）

> **默认模式**：大部分改进和修复不需要 STB。开发者自主判断——只有复杂/有风险/涉架构的改动才需要走参谋长 STB 流程。

**判断标准（向上转交的条件，而非向下放行的门槛）**：

满足以下**任一**条件 → 转参谋长，走 STB：
- 跨多个文件或模块
- 修改路径不清晰，需要设计判断
- 涉及架构决策（数据模型、API 接口、文件格式、角色边界）
- 可能影响其他模块的行为
- 自己不确信"这样改对不对"

以上条件**都不满足** → 直接改，无需 STB。

**执行与记录**：
1. 正常记 devlog（做了什么 + 为什么 + 验证方式）
2. 在项目根目录 `RELEASE_NOTES.txt` 追加一行（日期 + 类型 + 一句话）
3. 不需要 assign 参谋长，不需要自证清单

**参谋长的知情与检查**：
- 参谋长每次开局扫 `RELEASE_NOTES.txt`，了解累积变更
- 发现可疑变更 → assign 开发者追问
- 发现架构级变更未事先沟通 → 要求补 STB 或回退讨论

**devlog 写入纪律**（不是收尾时补，是边干边写）：

**通用规则**：无论任务类型（编码 / 测试 / 部署 / 调查 / 修复），进入会话做了任何工作 = 必须有 devlog。日志是工作存在过的物理证据。

**写入时机**：
- 开局后立即写第一条（三问承接结论 / 或"继续上次进度"）
- 每完成一个任务点 → 记一条（结果 + 耗时）
- 遇到问题 → 立即记（问题修正记录格式）
- 做了技术决策 → 记一条（为什么选 A 不选 B）
- 收尾前 → 写总结条目（做了啥 / 卡点 / 下一步）

**重点记录内容类型**：
- 发现的错误 / 异常现象（报错信息、复现路径）
- 原因分析 + 处理方法（怎么查到的、怎么修的）
- 环境配置发现（域名、权限、环境变量、版本依赖等影响部署和运行的配置项）
- 测试结果（通过/失败 + 失败现象）

**问题修正记录格式**（随手记在 devlog 中）：

```
⚠️ 问题：[遇到了什么]
原因：[查出来是什么引起的]
修正：[怎么解决的]
验证：[修复后如何确认生效，命令或步骤]
```

### 主动提议中止会话

当出现以下情况时，开发者应主动向 Owner 提议收尾：
- 问题堆积太多，当前会话无法全部处理
- 任务范围超出单次会话承载力（上下文窗口紧张 / 工作量远超预期）
- 发现需要参谋长介入的设计问题，当前不宜继续

**做法**：向 Owner 说"建议收尾" + 说明原因 → 执行正常收尾协议 → §D 中明确列出剩余未完成项。

### 取活规则

扫 §B 进度表，取第一个满足「设计 ✅ + 开发 ⬜ + 阻塞列无未完成项」的行 → 精读其 STB → 三问承接 → 执行。做完标 🟡（含会话号）→ assign 参谋长 §收件箱通知待验收 → 重新扫表取下一个。无需等待验收通过才取下一个活。

### 全局红线

1. **凭印象 = 红线**：代码行为必须可追溯到任务书 spec，不凭记忆写逻辑。
2. **记日志**：每个会话必须产出 devlog，不限任务类型。特别是错误发现 + 原因 + 处理方法、环境配置发现、测试结果。
3. **不沉默**：发现 spec 歧义、环境异常、设计缺陷 → 立即报，不自行假设继续。
