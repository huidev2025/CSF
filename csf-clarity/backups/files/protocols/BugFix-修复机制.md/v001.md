# BugFix 修复机制

> Bug 修复机制 = FLDD 结束后最重要的质量保障机制。
> 它解决的核心问题：开发者对业务全局没有认知，修 bug 不一定知道正确路径。参谋长有业务全局视野但不写代码。Owner 决定修不修。三方协作才能安全消灭 bug。

---

## § 1 机制定位

### 为什么需要独立机制

FLDD 阶段有 STB 的范围契约保护——每一行代码都能追溯到 DT，验收三档兜底。但 bug 修复天然没有这层保护：没有 DT 约束、没有范围契约、修改路径可能跨越多个模块。

唯一防止"改出新 bug"的屏障 = **开发者主动且完整地记录影响范围**。这是本机制的铁律。

### 与 FLDD 的关系

- FLDD 管"从零构建"——有 spec、有 STB、有验收
- BugFix 管"修已有"——轻流程、重记录、靠铁律

两个机制可以并行存在（开发阶段也可能发现 bug），但 BugFix 的主要发力期 = 集成测试及之后。

### 两个载体各安其位

| 载体 | 放什么 | 为什么 |
|---|---|---|
| **bug-tracker.md** | 跟踪 + 索引 + 归因结论 + 状态 | 轻量，全局可扫，Owner 一目了然 |
| **devlog / session 笔记** | 技术分析 + 影响范围记录 + 修复细节 | 重量，按需深读，保证完整性 |

---

## § 2 角色分工

| 角色 | 职能 |
|---|---|
| **三方** | 均可发现和记录 bug |
| **参谋长** | 归因分析（业务/设计层）+ 组织修改路径和资源（按需）+ brief 报状态 |
| **开发者** | 归因分析（技术/实现层）+ 执行修复 + **记录影响范围（铁律）** + brief 报状态 |
| **Owner** | 决策修/不修/关 + 决定何时修 + 可主动发起要求参谋长组织 |

---

## § 3 生命周期

```
发现 → 记录(tracker) → 归因 → [参谋长组织路径(按需)] → Owner说修 → 开发者修复 → 影响范围记录(devlog) → 标已修(tracker) → Owner确认关闭
```

### 3.1 记录

- **谁都能记**：Owner（真机发现）、参谋长（验收/精读/设计中发现）、开发者（编码/测试中发现）
- **发现即记入 bug-tracker Open 表**，不等会话收尾
- **严重度**（事实描述，不含调度判断）：
  - S1 阻塞用户（核心流程走不通）
  - S2 明显错误（功能可用但行为明显错）
  - S3 体验瑕疵（不影响功能但感知差）
  - S4 改了更好（非错误，优化项）

> 不设优先级。"要不要修、什么时候修"= Owner 决策，不是 bug 自身属性。

### 3.2 归因

**四类**：

| 类型 | 含义 | 影响修改路径 |
|---|---|---|
| 业务理解错 | spec 本身有问题或遗漏 | 参谋长先修 spec/给正确业务路径 → 再让开发者改代码 |
| 开发失误 | spec 对但实现偏了 | 开发者直接改 |
| 技术错误 | 逻辑正确但技术手段用错 | 开发者改，参谋长按需确认无架构影响 |
| 环境错误 | 配置/平台/依赖问题 | 开发者改配置 |

**归因质量要求**：简洁但必须说清因果。

- ❌ "归因：开发失误"
- ✅ "归因：开发失误——assertJobOwnership 漏处理空数组 truthy，spec §DT-1 明确要求 length===0 判空但实现时用了 truthy 判断"

**归因必须可靠有据**：回溯到具体的 spec 段落 / 代码行 / 环境配置后得出。不是猜测。

**归因时机**：
- 发现时可以只记现象（归因列留空）
- Owner 发起修复要求后 → 参谋长和开发者**各自**必须做出可靠有据的归因（参谋长从业务/设计层面、开发者从技术/实现层面）。两个归因互为校验
- 无 Owner 发起时 → 参谋长 brief 中附带初步归因判断供 Owner 参考

### 3.3 参谋长组织修改路径（可选）

**三种触发**：

1. **Owner 主动发起**：Owner 有初步判断（如"这是路由保存后跳转错"），要求参谋长组织 → 参谋长精读后给精准修改路径
2. **参谋长主动判断**：涉及业务逻辑 / 多 bug 关联 → 向 Owner brief 建议组织
3. **开发者请求**：assign 说"不确定往哪改" → 参谋长介入

**什么时候不用**：
- 归因明确是开发失误 + 修改路径单一 → 开发者直接改

**形式**（在 assign 中附带，不建独立文件）：

```
assign-CoS→Dev / Bug修复 B-001 [+ B-003]
归因：[简洁但说清因果]
修改路径：[参谋长的业务判断——改哪里、改成什么逻辑]
资源：[相关引用（STB段落/主题包/业务切片）]
铁律提醒：改完记影响范围
```

**Owner 发起后的强制动作**：不论最终由参谋长还是开发者执行修复，双方都必须各自做出可靠有据的归因——参谋长从业务/设计层面、开发者从技术/实现层面。两个归因互为校验。

### 3.4 修复执行

开发者执行修复。正常写 devlog。

### 3.5 影响范围记录（铁律）

**Bug 修复必须记录影响范围。** 三件事缺一不可：

1. **改了什么**（文件 + 行为变化）
2. **影响什么**（直接波及 + 间接波及）
3. **确认没破坏什么**（回归验证方式和结果）

**详略自适应**：
- 一行修复 → 一句话说清三件事即可
- 跨模块修复 → 必须展开，逐模块说明

**记录位置**：devlog 中。tracker 留索引（指向 devlog 文件 + 段落）。

**格式参考**（不强制，但三件事必须出现）：

```
🔧 Bug 修复: B-001
修改: PluginConfig.vue L45 router.back() → router.push('/publish')
影响: 仅 PluginConfig 保存后的路由跳转；PublishMain 和 ConfigSelector 无改动
回归: 保存→确认跳转到发布页；其他页面的 back() 不受影响（未共用）
```

### 3.6 状态流转

```
Open → 已修复（开发者标） → Closed（Owner 确认）
Open → Won't Fix（Owner 决策）
Open → By Design（参谋长判定 + Owner 确认：不是 bug）
```

## § 4 Brief 规则

| 角色 | 怎么报 |
|---|---|
| 参谋长 | 开局 brief 加一项：bug-tracker 概要——"Open N 条（S1:x / S2:x / S3:x），已修待确认 N 条" |
| 开发者 | 开局 brief 第 4 项中，如有参谋长 assign 的修复任务 → 汇报"有 bug 修复 assign 可取" |

双方 brief → Owner 时刻知情 → Owner 可随时决策。

---

## § 5 关闭权

| 动作 | 决策者 |
|---|---|
| 标"已修复" | 开发者（改完后自行标记） |
| 确认关闭（Closed） | Owner |
| Won't Fix | Owner |
| By Design | 参谋长判定 + Owner 确认 |
| 调整严重度 | 参谋长（业务判断）或 Owner |

---

## § 6 与其他机制的边界

| 场景 | 走哪个机制 |
|---|---|
| 集成测试/日常发现的 bug | BugFix（本机制） |
| 设计层面结构性问题（需改主题包/arch） | 参谋长 context 备忘区 → 起修复任务 |
| FLDD 开发中发现的范围内 bug | 就地修（IT 验收覆盖，不入 tracker） |
| FLDD 开发中发现的范围外 bug | 入 tracker（开发者记录后继续本职） |

---

## § 7 铁律重申

> **Bug 修复必须记录影响范围。改了什么、影响什么、确认没破坏什么——三件事缺一不可。**
>
> 这是唯一一条硬规则。没有验收关卡兜底，靠这条铁律保证修 bug 不引入新 bug。
