# 哨兵锚点法（HTML 注释定位提高 replace 效率）

## 工艺

在需要频繁搜索/替换的文件中，预埋 HTML 注释格式的锚点标记（哨兵），用于精准定位 replace 操作的 oldString。

## 为什么需要

session-171 实测：收尾时超 50% 时间耗在「构造长 oldString → 失败 → 重试」。`replace_string_in_file` 要求字面精确匹配——长 oldString 对全角空格、emoji、链接形态极敏感，一个字符差异就失败。

**哨兵解决方案**：用 30-40 字符的唯一锚点字串替代动辄几百字符的内容匹配，命中率从反复重试变为一次成功。

## 三种锚点形态

| 形态 | 格式 | 用法 |
|---|---|---|
| **包围对** | `<!-- @ANCHOR X -->` … `<!-- @/X -->` | 整段重写：oldString = 锚点对 + 旧内容 / newString = 锚点对 + 新内容 |
| **尾哨兵** | `<!-- @APPEND:X ... -->` | 追加：oldString = 单行哨兵 / newString = 新内容 + 同一哨兵 |
| **界哨兵** | `<!-- @TAIL:X ... -->` | 删尾/截断：定位保留区与待删区的分界 |

## 三类替换模式

**追加型**（ROI 最高）：
```
oldString = <!-- @APPEND:SEC-2.0-D ... -->
newString = - 新教训条目\n<!-- @APPEND:SEC-2.0-D ... -->
```

**整段重写型**：
```
oldString = <!-- @ANCHOR X -->\n旧内容\n<!-- @/X -->
newString = <!-- @ANCHOR X -->\n新内容\n<!-- @/X -->
```

**删尾型**：
```
oldString = <!-- @TAIL:X -->\n待删段
newString = <!-- @TAIL:X -->\n
```

## 纪律

- 锚点只用于 replace 定位，渲染时不可见（HTML 注释）
- oldString 超过 100 字符 → 考虑改走哨兵路径
- 必须长字面时 → 先 read_file 取最新字面（防过期匹配）
- 新加章节同步追加哨兵 / 删除章节同步移除哨兵
- 不在具体 bullet 之间放锚点（会随增删漂移）

## 适用场景

任何需要频繁 replace 的长文件（CONTEXT / 大型 README / 滚动日志）。特别是含中文、emoji、全角标点的文件——哨兵将匹配复杂度从 O(内容长度) 降为 O(锚点长度)。
