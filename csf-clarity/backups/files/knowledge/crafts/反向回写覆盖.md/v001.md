# 反向回写覆盖（改了 A 必须同步所有引用方）

## 工艺

修改任何文件内容后，必须检查并同步所有引用该内容的位置。方向不只是"下游"——上游、平级、索引都可能引用。

## 四方向检查清单

| 方向 | 典型位置 |
|---|---|
| 上游 | arch 文档、domain spec、全局共识层 |
| 下游 | 主题包 package/sources/verify |
| 平级 | 跨包 cross-ref、责任分隔对偶包 |
| 索引 | GP 主表、候选清单、主题分包清单、来源 Slice §1.6 |

## 原因

TA-08 消费者测试暴露：T3 阶段把内容从 arch 分拆搬运到主题包后，arch 原文未做反向修订（A1/A2/A3 全是此类）。结果：同一事实在不同文件中表述不一致，下游开发者读到过时版本。

**根因**：人天然关注"我改的文件"，不主动想"谁引用了我改的内容"。

## 防御方法

改完文件后追加一步：**grep 关键标识符**（字段名/枚举值/接口名/spec编号）→ 列出所有命中文件 → 逐个检查是否需要同步。

## 自检问句

- 这个改动有没有被别的文件引用？
- 如果我是另一个文件的读者，读到旧版本会怎样？
- 有没有反向 cross-ref 需要补？

## 延伸：复查后必须评估影响范围并同步

遗忘某个已完成的事实是正常的（AI 无跨会话记忆）。但**经过提醒并复查落实后**，不能只回答"对，做过了"——必须：

1. **评估影响范围**：这个已完成的事实，哪些文件还在引用旧状态？
2. **当场同步**：修正所有过期引用，使文档体系回到一致状态
3. **目的**：避免下次再忘 → 再搜索 → 再复查同一件事

> 忘了 → 被提醒 → 确认 → 评估影响 → 同步修正。这是完整闭环。只做到"确认"而不做"同步"= 下次还会踩同一个坑。
