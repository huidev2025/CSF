# 印象连锁警觉 — 发现一个错误后立即扩大检查面

## 原则

当自我纠错发现一个凭印象错误后，**不停止于修复单点**，立即触发反射：**"我关于这件事的其他断言，是不是也凭印象？"** ——重读所有相关源文件。

## 原因

AI 的凭印象错误极少孤立出现。一次凭印象往往会产生连锁误判。如果只修复发现的一个错误而不检查其他断言，剩下未发现的会流入下游（设计文档/STB/代码），由后续环节抓出——浪费多轮通信，侵蚀信任基线。

典型模式：发现了 A 文件的函数签名写错 → 纠正了 → 但没有追问「我关于 B 文件、C 模块的其他断言是不是也凭印象？」→ 其余错误继续传播。

## 防御方法

| 时机 | 动作 |
|---|---|
| 发现自己凭印象一次 | 立即列清单：「本次会话我还做了什么关于代码/文件状态的断言？」 |
| 清单中每一项 | 重读对应源文件，逐条确认或修正 |
| 涉及写入 STB/设计文档 | 检查所有 API 引用是否与源码一致 |
| 确认完毕 | 在 session log 中注明「印象连锁检查完成，共修正 N 处」 |

**自查口诀**：纠错了一个 → 问「还有没有？」→ 重读 → 确认。

## 案例概述

| # | 会话 | 触发场景 | 简述 |
|---|---|---|---|
| 1 | csf-v4-live 027 | STB API 描述与实际代码不符 | 发现某文件函数签名写错后纠正，但未扩大检查。导致 10 处 API 差异流入 STB，开发者三问时全部抓出 |
