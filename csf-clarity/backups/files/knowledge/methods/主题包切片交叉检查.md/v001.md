# 主题包定稿后须做切片交叉检查

## 教训来源

session-373 发现 job 页"转发概况栏"从未被实现（M4 F1.6）。根因是：

- TD-02（小程序壳主题包）定稿于 session-138（2026-05-19）
- 主-5（转发节点切片）立项于 session-168（2026-05-21），晚 30 次会话
- TD-02 信息来源仅为 arch 层导航状态机，arch 层不定义组件渲染规格
- 主-5 立项后无"回检已定稿主题包"机制
- STB 设计时只看到导航触发器，不知道组件需要建造

## 原则

**主题包定稿时序无法保证"完备"——必须在 STB 设计阶段做切片交叉检查。**

STB 草稿产出后，强制对照相关业务切片（主-N / 插-N），检查是否有切片补充的 UI 组件/数据字段在本 STB 中被遗漏。

## 操作方式

| 步骤 | 动作 |
|---|---|
| 1 | 列出本主题包涉及的所有业务切片（按 doc/domain/ 清单） |
| 2 | 逐切片扫"§N.M 渲染规格"，列出组件名 |
| 3 | 对照 STB 中的 DT 清单，标记是否已覆盖 |
| 4 | 未覆盖 → 追加到 STB 或单独立 IT |

## 反模式

- 只看 arch 层（导航状态机/组件图）定义主题包 → 漏渲染规格
- 用"切片不覆盖 UI 细节"声明逃避责任 → STB 反而更需要主动查
- 假设"切片和主题包同步定稿" → 实际时序经常差距 30+ 会话

## 关联计划

tp-373 主题包×切片扫描：对所有已定稿主题包做一次补偿性扫描，产出缺口清单。
