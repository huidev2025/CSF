# 计划符号必须唯一且不可变

## 教训

session-330 定了 T1~T6 共 6 项任务计划。session-332 开局时：
1. 凭印象插入了一个 T0（打乱全部编号）
2. L2 brief 中另起 ①~⑥ 表达"本次会话步骤"
3. Owner 纠正后又改为 1~4 步
4. 最终整个会话花在对齐"我到底在说哪个任务"上，零产出

## 根因

- **收尾时没把计划的准确编号和内容记清楚**（只记了大意，下次凭印象还原时就变形）
- **开局时没忠实还原上次计划，而是"重新规划"**（跨会话计划存在于 README，不存在于参谋长脑中——必须真读后按原文执行或显式声明变更）
- **符号链接越多越乱**：用两套编号指代同一组任务 = 制造符号链接。每多一层映射，就多一个出错点

## 正确做法

1. **上一会话有结论就记笔记**——结论含计划时，计划的编号/内容必须 verbatim 记入 README
2. **收尾时读笔记，总结结论**——确保 README 的跟踪表与会话中实际决策一致
3. **新会话发现计划需变化 → 先讨论变更再落地**——说清楚"我认为原计划的 T3 应该拆成 T3a+T3b，原因是..."，Owner 确认后更新 README。不是直接在 brief 中用新编号覆盖
4. **一个任务只有一个编号**——不允许同时存在 T2、②、"第四步" 三个符号指向同一件事
5. **README 是唯一权威**——brief 引用计划时直接说"T2"，不另起符号

## 概括

**跨会话跟踪的计划一旦落地到 README，编号和内容就是锚。任何变更必须显式讨论后更新 README，不可在 brief 中隐式创造替代编号。**
