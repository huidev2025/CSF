# happy path 偏好（AI 天然盲区）

## 方法

AI 在设计/分析/起草时天然只覆盖「主线 + 差异化」，而「犹豫/取消/失败/边界/终态 GC」整片被默认丢给"以后再说"。这不是偶发疏忽，是结构性偏好——必须主动对抗。

## 原因

TA-08 消费者测试（session-152/153）中 20+ 条回问清单，大部分指向「非主线路径」：
- 事务回退（支付后创建失败怎么办）
- 状态转换边界（ongoing vs published 落库时机）
- 资源清理（过期草稿/废弃 token/断连恢复）
- 取消流（用户中途退出/发布者撤回）

这些不是"低优先级细节"——它们是系统健壮性的主体。happy path 只占真实运行时间的一部分。

## 对抗方法

| 时机 | 动作 |
|---|---|
| 设计时 | 对每个主线步骤问：如果这步失败/中断/超时/用户反悔会怎样？ |
| 起草 spec 时 | 专设「异常/边界/终态」章节（不是附录而是正文） |
| 验收时 | 用"红队四视角"（4-Lens ABSC）生成非 happy path 场景 |
| 自查时 | 检查产出物中 happy path vs 非 happy path 的篇幅比——失衡即预警 |

## 自检口诀

写完一个流程后问：**如果这里断电/用户关页面/网络超时/数据不一致——系统处于什么状态？** 答不上来 = 盲区未覆盖。
