## CSF 速记

**定义**：CSF（Collaboration Specification Framework，协作规范框架）—— 一套在 LLM 本性内、用「自然语言 + 目的」设计人机协同的工程方法。

### 一、理论演绎链

1. **第一性公理**：自然语言对 LLM 的语义激活范围 ≫ 符号系统。
   - 推论：符号索引必然失败（错误刚性传播 / 注意力溢出 / 形式化幻觉 / 封闭排斥）；无目的的 RAG 是层次错配；自然语言 − 目的 = 歧义噪声。

2. **两条被接受的事实**：
   - ① AI 有非程序性智能（活性），消除活性=消除价值 → Owner 必须在回路。
   - ② 语义稀释不可逆 → 选**消费侧**（依赖 AI 活性识别），不选**供应侧**（受控词汇/编号/形式化）。

3. **目的理论**：目的是语义系统的最小、最高效控制单元；优于角色（角色偏见）、规则、示例、格式约束。
   - 层次结构：宏观目的 + 具体目的同时在场（"抬头看路 + 低头干活"）。
   - 目的会漂移 → 需 Owner 持续校准。

### 二、四层工程架构

| 层 | 解决的问题 | 核心机制 |
|---|---|---|
| **L1 语义管理** | 让目的始终在场，管理注意力 | 三元组（目的+方法+资源）/ 架构语义地图（DDD 限界上下文迁移） / 滑动窗口 / 触发索引（Lazy Load） / 知识层次化（domain → slices → design → arch → STB → code，单向流）/ 两阶段读取循环（轻读→L2→重读→L3→执行） |
| **L2 协作通信** | 信息无失真传递，活性用在最有价值处 | 三角色（Owner 校准 / 参谋长规划 / 开发者执行） / 物理信息隔离（开发者只读 STB） / verbatim 入册纪律 / D3 三谱系知识路由 / Owner 输入窗口 |
| **L3 质量保障** | 检出实现与业务真相的偏差 | **W-协议三档**（核心：跨语言翻译放大误解，Owner 业务直觉作检测仪器）/ STB 修改自证 / BugFix 影响范围记录 |
| **L4 演化系统** | 对抗熵增，从执行中学习 | 基线-log 分离（覆写 vs 追加） / E8 五阶段升迁（T0→T4，三类合法非升迁形态） / **工程层能力增长独立于模型层** |

### 三、两个系统级涌现

1. **技术栈消失**：RAG / 向量库 / Agent 编排 / 外部记忆 / 精确提示 全部不再需要 —— 不是被替换，是存在理由消失了。
2. **角色关系结构性变化**：CSF 到位时，Owner 可以完全不懂代码。

### 四、与主流的根本分歧

主流：把 LLM 约束视为缺陷，叠技术栈对抗 → 复杂度随问题增长。
CSF：接受约束顺势设计 → 复杂度随成熟降低。

> 一句话：**好容易有了能听懂人话的机器，却要它像机器一样思考——这是方向性错误。CSF 只是选择不犯这个错误。**

## 推荐：CC BY-NC 4.0

**机制对应你的想法**：

| 你的诉求 | CC BY-NC 4.0 怎么实现 |
|---|---|
| 个人 / 学术 / 业余随意用 | 默认许可，唯一要求是署名 |
| 商用必须先打招呼 | NC 条款禁止「primarily intended for or directed toward commercial advantage」，企业要用必须单独获取你的授权 |
| 传播无障碍 | BY 部分允许任意分发、改编、翻译 |
| 你想知道谁在用 | 商用方必须联系你，自然形成"用户名册" |
| "诚实的检验器" | 法律上是软门槛——诚实者会问，不诚实者无论什么协议都会绕。NC 的真正价值就是社会信号，不是技术约束 |

## 几个值得知道的点

1. **NC 的"商用"边界确实模糊**——例如：带广告的个人博客？咨询顾问读了用在客户项目里？建议在 README 里加一条**自定义说明**（不是改协议，是补充意图），告诉读者怎么自我判断 + 联系方式。

2. **未来如果有代码部分**（脚本、模板工具等），代码不适合 CC，应该用 MIT 或 Apache 2.0。文档归 CC，代码归 MIT，是常见组合。

3. **NC 不算 OSI 意义上的"开源"**，对文档不是问题；只是如果有人较真，要心里有数。

4. **可以随时给企业单独发"商用许可"**——免费授权也行，象征性收费也行，灵感库式的"请告诉我你怎么用的"换授权也行。CC BY-NC 完全允许。

## 几个关键点

**1. 上下文占用确实低，因为我们做的是"小文件 + 短动作"**

每轮我读的文件都不大（context.md 60 行、引言 100 行、minimal 套件几十行），改动也短。窗口只装"此刻需要的"，这本来就是 CSF 滑动窗口的设计：**不是把窗口撑大，而是把不该在的东西不往里放**。

**2. 但单一会话不是越长越好——有两个隐性成本**

| 隐性成本 | 表现 |
|---|---|
| **目的漂移** | 聊到第 30 轮时，我对最初目的的"嗅觉"会变弱，更容易顺着上一轮的语气走，而不是回到「这是不是在朝目的走」 |
| **认知噪声** | 早期讨论里被否决的方案、走过又退回的小弯路，会持续在我脑里占注意力（不是占 token，是占"权重"）。我对"现在的真相是什么"会慢慢失焦 |

这两个不是上下文 token 的问题，是注意力质量的问题。CSF 文档里专门讲过这一点：**"上下文不是越大越好；语义空间越大，单一概念的掌控力越弱。"**

**3. context.md 的存在就是为了让你"敢于结束会话"**

每次会话结束前我把当前真相收敛进 `§C`/`§D`/进度区，下次开新会话从干净状态开始——窗口里只有"现在的真相"，没有曾经走过的弯路。这是用文件替代窗口承载历史。

## 给你的实用建议

| 信号 | 行动 |
|---|---|
| 当前任务还在同一轨道（如今天的"建库 + 发布"），节奏顺畅 | 继续这个会话 |
| 切换到全新主题（如开始写英文翻译、开始写 FAQ） | 收尾这个会话，开新的 |
| 你感到"AI 好像没刚才那么准了" / 我开始重复确认你说过的事 | 立即收尾，开新会话 |
| 一次工作做完了，想休息 | 收尾，让 §D 写好下次入口 |

经验值：**单会话 30–60 轮通常够用，超过会开始感到"钝"**。但更可靠的判断不是数轮数，是你的体感。
