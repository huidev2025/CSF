# 修复原则（说原因不说行号 · 影响大则先 brief Owner）

## 适用场景

- 参谋长在 STB 中写修正类指令
- 开发者自行发现 bug 并修复
- 任何角色发现代码/spec/配置中的不一致需要修正

## 铁律（四条）

1. **先搞清楚历史背景、原因、修改原则和方法**
2. **记录历史，记录解法分析决策**（落文件，不在会话中空转）
3. **形成方案或指令时，说明原因、目的、原则和方法** — 不能只给具体字段或具体行号的修改要求
4. **避免自己的修改带来新的污染**（覆盖不全、遗漏新增位置、引入不一致）

## 影响面评估 + Owner brief

修正问题前必须评估影响面：
- **影响面小**（单文件、孤立 bug）：自行修正 + 记录即可
- **影响面大**（跨多文件/模块、模式性问题、需要回溯历史才能判断正确状态）：**主动给 Owner 进行 brief**——说清原因、影响范围、解决方法的选项，等 Owner 确认后再动手

> 为什么 brief：影响面大的修正往往涉及设计决策、历史背景。Owner 可能知道你不知道的背景；Owner 也需要对重大变更有知情权和否决权。

## 为什么不给行号

1. **执行者有理解能力** — 理解原因和动机，就能覆盖所有需修正的位置（包括你没看到的、新增的）
2. **情况会变** — 文件/行号可能已因并行工作而不存在
3. **责任链** — 给原则 = 执行者对结果负责；给行号 = 执行者只对复制粘贴负责
4. **避免新污染** — 替换清单本身可能覆盖不全

## 方法（修正类问题的标准动作）

1. **溯源**：这个错误/不一致最初怎么产生的？什么时候产生的？
2. **查同类**：是模式性问题（多处）还是孤立（单点）？
3. **评估影响面**：小影响自行处理；大影响 → brief Owner
4. **记录**：在 STB / devlog / assign 中写下历史背景 + 正确状态的判定依据
5. **原则式方案**：
   - ✅ 原因是 A，正确状态应该是 B，判定原则是 C，验证方法是 D
   - ❌ 把第 37 行的 `signUpScene` 改为 `signUp`
6. **给验证标准**：告诉如何验证"做完了"（如 grep 零命中），而非列出所有要改的文件

## 案例

**反面**："修改 pluginDescriptors.js 第 42 行 + createJob/shared/ 第 42 行 + confirmJob/shared/ 第 42 行..."
→ 遗漏了新增副本、行号与实际不符时无从判断。

**正面**："历史：018 spec v0.1 原名 signUpScene...正确状态：所有 dev/ 下 pluginId 应为 signUp...原则：grep 全扫...验证：grep -r 'signUpScene' dev/ = 0 命中。"
