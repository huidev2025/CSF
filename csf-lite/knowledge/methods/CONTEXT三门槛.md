# CONTEXT 三门槛（防引擎膨胀）

## 方法

写入 CONTEXT（E1 引擎层）的内容必须同时满足三个门槛：

| 门槛 | 含义 |
|---|---|
| **高价值** | 对工作质量有显著影响 |
| **高频** | 几乎每次会话都会触发/用到 |
| **已沉淀** | 经过实践验证、不再是试验性的 |

**任一门槛不过 → 走其他承载位**（守则 / 协作规范 / 方法论库 / inbox / exec-log / 笔记）。

## 原因

CONTEXT 是 AI 每次启动必读的文件。它膨胀 = 每次启动负担加重 + 重点被稀释 + 真正高频的条目被埋没。精练的 CONTEXT 执行力 > 冗长的 CONTEXT 覆盖面。

## 反例

session-199：IT-2 范围爆炸后，一度将 200+ 行详细范围描述压入 CONTEXT。不过"高频"门槛（仅该 IT 期间有效），应放在 TP/STB 层级。

## 层级判断指引

| 内容性质 | 推荐承载位 |
|---|---|
| 每次会话必触发的引擎规则 | E1 CONTEXT |
| 跨任务通用原则 | E2 守则 |
| 角色间接口协议 | E5 协作规范 |
| 具体操作手法 | E6 方法论库 |
| 单任务期间有效的约定 | TP README / STB |
| 临时性待办 | inbox / PENDING |
