# UI 密集型 STB 验收三要件

## 原则

UI 密集的 STB（DT 数量 ≥10）在验收时必须满足三个要件，不可全部依赖弱档截图验收。

## 原因

弱档验收的本质缺陷：
- 截图只能判断「有没有这个控件」，无法判断「文本格式/颜色值/按钮位置/间距是否正确」
- 验收者注意力天然聚焦于功能正确性（点了有没有反应），而非设计保真（颜色/尺寸/布局）
- 单会话验收 DT 过多 → 注意力分散 → 细节遗漏率急剧上升

典型案例：18 个 DT 全部弱档 + 单会话验收 → 多处视觉差异（标注行缺说明、替代控件形态、缺样式约束、内容缺失）全部漏过。

## 操作手法

### 要件一：档位不能全弱档

UI STB 的 DT 档位配比建议：

| DT 类型 | 建议档位 | 示例 |
|---|---|---|
| 布局骨架 | 🟡弱档 | 启动→截图确认布局 |
| **交互闭环** | 🔵中档 | 模拟数据→验证渲染输出（文本/颜色值/控件可见性） |
| 数据流正确性 | 🟢强档 | 构造测试数据→调用接口→断言输出字段值 |

DT 总数 ≥10 时，**至少 30% 应为中档或强档**（向上取整）。

### 要件二：单会话验收 DT 上限 ≤15

一个会话验收 DT 数超过 15 时，应拆分为两个验收会话：
- 第一个会话验收核心功能 DT
- 第二个会话验收边界/弹窗/整合 DT

不在同一会话中追求「一次性全部验收通过」。

### 要件三：验收增加设计保真维度

验收 checklist 中增加设计保真检查项（至少覆盖以下三类）：

| 检查维度 | 检查项示例 |
|---|---|
| **文本格式** | 标注文本是否与设计文档逐字一致 |
| **颜色/样式** | 色值是否匹配设计、线条宽度是否正确 |
| **控件形态** | 控件类型是否正确、是否缺约束（置灰/tooltip） |
