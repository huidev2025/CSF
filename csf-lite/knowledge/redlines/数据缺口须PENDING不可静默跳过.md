# 数据缺口须 PENDING，不可静默跳过

## 原则

开发者在实现 STB DT 时，若发现所需数据在当前作用域内不可获取，**必须 PENDING 回参谋长**，不可自行简化或静默跳过。静默跳过 = 设计→实现偏离的第一大来源。

## 原因

典型实现偏离模式：开发者需要的数据拿不到 → 选择了跳过而非追问 → 设计细节被静默违反：

| 偏离类型 | 缺失数据 | 开发者常见做法 |
|---|---|---|
| 保护机制未实现 | 锁定项清单 | 跳过，仅保留最简约束 |
| 标注信息缺失 | 目标项的元数据字段 | 跳过，仅显示标识符 |
| 导出内容不全 | 完整数据内容 | 跳过，仅导出概要 |

共同模式：遇到数据缺口 → 降低实现标准 → STB 规定被静默违反。参谋长在验收时看到「功能没报错」，但看不到「设计细节被省略」。

**FLDD 三栏自检本应拦截这类问题**：🟡不确定 → PENDING 回参谋长。但开发者有时会跳过三栏自检直接实现。

## 防御方法

| 角色 | 防御动作 |
|---|---|
| **开发者** | FLDD 三栏自检中 🟡不确定 / 🔵准备假设 非空 → **不动键盘**，PENDING 回参谋长。典型触发信号：「STB 要求 X，但我不知道怎么拿到 X 的数据」 |
| **参谋长** | STB 中涉及外部数据源的 DT，增加「数据可及性」检查：该 DT 需要的数据在当前代码中是否已有接口可获取？若否 → DT 中注明数据来源路径 |
| **验收** | 对涉及外部数据源的 DT，验收时不仅检查「有没有这个功能」，还要追问「实现中实际用了什么数据」——对比 STB 规定的数据源与代码中实际读取的数据源 |

**自查口诀（开发者）**：STB 说要做 X → 我能拿到做 X 需要的数据吗？→ 不能 → PENDING，不写代码。
