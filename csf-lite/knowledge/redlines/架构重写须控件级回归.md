# 架构重写须做符号级回归

## 原则

代码架构重写完成后，**必须逐符号核对**：每个被引用的函数、变量、组件，在新代码中确实存在。不可假设「肯定在」。

## 原因

架构重写时，构造/初始化代码被整体替换，旧符号可能被移除/重命名/合并。但散布在各处的调用方仍引用旧符号名。语言/框架可能在构造阶段不报错——只有在运行时触发该路径时才会出错。

更致命的是连锁效应：如果异常处理器也引用了缺失符号，第一次错误的 except 块会抛出第二个未捕获的错误 → 进程直接退出，日志中无任何 trace。

## 防御方法

| 时机 | 动作 |
|---|---|
| 架构重写完成后 | 列出所有新代码中定义的符号（函数/变量/组件） |
| 对照检查 | 搜索所有旧符号引用，交叉比对——引用但未定义 = 遗漏 |
| 发现引用无定义 | 补建符号，或修正引用 |
| 异常处理器 | 特别检查错误处理路径中的操作——它们是最危险的二次引爆点 |
| 回归测试 | 覆盖每个符号的触发路径（正常路径 + 异常路径） |

**自查口诀**：架构重写完 → 列出定义 vs 引用 → 交叉比对 → 异常路径单独过一遍。

## 案例概述

| # | 会话 | 触发场景 | 简述 |
|---|---|---|---|
| 1 | csf-v4-live 045 | UI 面板重写后崩溃 | 某状态标签被 3 处引用（双击发送、右键发送、异常处理），但重写后的构造代码中从未创建。连锁崩：第1次错误 → except → 第2次错误（标签不存在）→ 进程退出，日志无 trace。多轮调试在错误方向试错 |
