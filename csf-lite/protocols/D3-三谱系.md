# D3 三谱系（业务规则明文化 / 认知校准 / UI 显化）

> 所有"业务相关 D 行 / Q 项"必属三谱系之一。三谱系决定**落点**（写到哪个文件）+ **触发动作**（要不要回写 spec / domain）+ **W 档**（W-轻可关闭 vs 必走 W-重）。
>
> 元约束：[守则 § 1 业务唯一裁判](../守则.md)。

---

## § 1 三谱系定义

| 谱系 | 本质 | 落点（主） | 落点（从）| W 档 |
|---|---|---|---|---|
| **谱系 1 业务规则明文化** | Owner 补齐了原本含糊 / 未表达的业务规则 | `doc/domain/` 对应业务文件 | DEVNOTES D 行索引 | W-轻 即可（已是 Owner verbatim） |
| **谱系 2 认知校准** | AI / 团队对业务的理解**偏离了真相**，被 Owner 纠偏 | DEVNOTES D 行（B 型，verbatim 入册）| 影响的 spec / domain 同步修订 | **必走 W-重**（业务认知偏差只有 Owner 在场能放大检验）|
| **谱系 3 UI 显化** | 业务规则已稳定，但 UI 表达 / 文案 / 交互层显化不足 | `doc/design/` UI 设计文件 | DEVNOTES D 行索引（如属架构性 UI 决策）| W-轻 即可 |

---

## § 2 判别问题

接到一条业务相关 Q 项 / D 候选时，按以下问题判别谱系：

| Q | 是 → 谱系 |
|---|---|
| 这是 Owner 第一次说出这条业务规则吗？且与已有文档不冲突？| → 谱系 1 |
| AI / 团队对该业务的现有理解被 Owner 否定 / 纠正？| → 谱系 2 |
| 业务规则没变，只是改了"用户看到的样子 / 文案 / 流程顺序"？| → 谱系 3 |
| 同时属多谱系？| → 拆 Q：谱系 1 / 2 部分先入册；谱系 3 部分另立 |

## § 3 三谱系典型案例

### § 3.1 谱系 1 — 业务规则明文化

- **D-05 派生树只存直接子节点**：Owner 拍板"不要存 ancestorJobIds，做可视化时另立中间表"——业务规则首次明确。
- **联合发布草稿四态**：Owner 第一次描述 editing / waitingConfirm / confirmed / rejected 流转——以前没写过。

→ 落点：`doc/domain/` + DEVNOTES D 行索引；W-轻 收尾即可。

### § 3.2 谱系 2 — 认知校准

- **D-102 / D-105 / D-106 类**：AI 对插件协作模式 / 计费触发 / 应答交互的理解有偏差，被 Owner 纠正。
- **TA-08 W-重 暴露的红线**：「默认入口 = 发布页非广场页」「联合发布资金流不代收代缴」——AI 之前理解有偏。

→ 落点：DEVNOTES D 行 verbatim + 影响的 spec / domain 全部同步修订；**必走 W-重**；可能需"全库 Phase C 复检"（如 TA-08）。

### § 3.3 谱系 3 — UI 显化

- **D-XX 全局 UX 底部浮层替代详情页跳转**：业务规则不变，UI 表达层升级。
- **资源卡片整块可点击 / 右上角红 × 取消选择**：交互层显化。

→ 落点：`doc/design/UI/`；W-轻 即可。

---

## § 4 谱系混淆的常见错误

| 错误 | 后果 | 修正 |
|---|---|---|
| 把谱系 2 当谱系 1 处理 | 漏走 W-重 / 影响的 spec 没回写 / 留下业务认知偏差 | 严格走 W-重 + 全 spec 影响面扫描 |
| 把谱系 3 当谱系 1 处理 | 业务文件被 UI 细节污染 / domain 与 design 边界模糊 | UI 显化只入 design，业务文件保持稳定 |
| 把谱系 1 当谱系 2 处理 | 大量 W-重耗 Owner 时间 / 增量低 | 先 W-轻入册；如后续发现冲突再升 W-重 |
| 三谱系都不归（"挂账"）| 业务输入丢失 / 后续会话失忆 | 走 [Owner-业务输入窗口.md § 3](./Owner-业务输入窗口.md) 三处承载强制落位 |
