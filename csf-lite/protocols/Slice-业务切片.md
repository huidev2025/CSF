# Slice 业务切片方法论（v3 BC 前防漂移）

> **一句话**：**在原则约束下，把业务可能的所有变化穷举推演一遍，并固化成可被开发者机械查表使用的产物**，作为 BC 任务书的业务规则前置稿。

---

## § 0 定位

Slice 集合在 v3 总设计框架中**与 domain、UI 平级**，作为 arch 的三个输入之一：

```
doc/domain/  (原则源 / 不可变 / 不越改越长)
     ↓
doc/slices/  (实例层 / 业务切片 / 横向穿透 / 收敛信号)  ← 新立
     ↓
doc/design/UI/ (可见物 / 页面+要素+跳转)
     ↓
三者合流 → plan/arch/ → plan/tasks/ (BC) → dev/
```

**反向回流唯一允许的口子**：发现 domain 表述**错误**时触发 Owner 仲裁修订。**新词汇/补充细节一律入 Slice，不回写 domain**（domain = 不可变原则源）。

### § 0.1 权威层级（D-126 立 / 第一五七次会话拍板）

| 层级 | 角色 | 职责 |
|---|---|---|
| **最高权威** | **doc/domain/** | 业务定义的源；绝大部分原则已在 domain 中声明 |
| **设计师 / 领域分析师** | **AI（参谋长）** | 参与业务理解、提取原则、推演变化、暴露缺口、强制反向引用 |
| **检测器 / 仲裁者 / 补充者** | **Owner** | 指出与业务定义/定位不符之处、对原则做补充判断、仲裁冲突 |

**纪律**：
- 除非发现 domain **错误**，否则一律以 domain 为准
- AI **不是任何一方的传声筒**；AI 提取原则后必须反查 domain 验证；Owner 补充原则后 AI 必须反查 domain 校核
- **Owner 不是唯一权威**；Owner 与 AI 的共同工作是「发现 → 引用 → 必要时补充 → 反查校验」

## § 1 Slice 必做三件事（D-123 Owner 拍板）

> **Slice 的目的（D-126 重述）**：**在原则约束下，把业务可能的所有变化穷举推演一遍，并固化成可被开发者机械查表使用的产物**。三件事是支撑这一目的的最小动作集。

### § 1.0 使用受众与两层目的（D-128 补遗 / 2026-05-20 第一五九次会话 / Owner verbatim 补入册）

> **背景**：D-124 拍板 α 双形态时 Owner 主张过「A 辅链接的定位」与「Slice 给开发者看的两层目的」，当时未记入册。D-128 补遗。

**Slice 的两层使用受众 + 两层目的**：

| 受众 | 主要使用时机 | 访问目的 |
|---|---|---|
| **参谋长**（第一手消费）| 写 BC 任务书时 | 速查结论 + 作为 BC 的业务依据 |
| **开发者**（第二手消费）| 拿到 BC + Slice 索引后 | **① 帮助理解业务**（横向穿透，避免自行从 domain 重做设计）**② 速查结论**（机械查表，无需层层翻 domain）|

**关键原则**：
- **避免开发者从 domain 重做一遍设计**。完成 Slice 本身就是为了让开发者不需要重做业务推演。
- **A 辅链接 = 证据来源，不是给开发者回查 domain 用的**。必要时才回查；主要使用方式是直接从 B 主表得到结论。
- **开发者要自己理解业务**（这点与§4.2 F1/F2/F3 价值一致），但理解的入口 = Slice 帮他预进，不是「domain 重读」。

## § 2.0 工作流程（D-126 新增 6 步 + D-130 新增前置零步 / 每篇 Slice 强制 / 不简化、不并步）

> **来源**：D-126 第一五七次会话拍板 6 步显式化 / D-130 第一六二次会话补前置零步「按目的备料」（统一解 D-122/123/124/130 历次幻觉循环根因 = §3「资源」项长期不执行）。

### 第 ⓪ 步：按目的备料（D-130 立 / 强制前置零步 / 反幻觉根因柱）

> **触发**：第一六二次现场实验三事实并列暴露——能力够 + 资料够 + 失败=不查权威定义凭印象推理。本步是其它所有步骤的**前置条件**：未完成不进入第①步。

**动作**：

a. **微观目的一句话陈述**：本 Slice 要回答什么具体业务问题（不抽象、不"验证方法论"等空头目的）  
b. **资料模块清单**（按类型分 / 不限于业务名词 / 见 [守则 §3 真知而非印象](../守则.md)）：

| 类型 | 默认权威位置 | 本切片需读章节 + 预期获得 |
|---|---|---|
| 业务名词 | [doc/domain/001 词汇表](../../doc/domain/001-domain-词汇表.md) | （按切片实例填）|
| 业务规则 | [doc/domain/](../../doc/domain/) 各业务节 | （按切片实例填）|
| 设计原则 | [plan/arch/](../../plan/arch/) / [DEVNOTES](../DEVNOTES.md)（新）+ [plan/DEVNOTES.md](../../plan/DEVNOTES.md)（D-01~D-141 归档） | （按切片实例填）|
| 前期设计 | 已有 Slice / 已有 TP / spec / 历次 D 行 | （按切片实例填）|
| UI 准确描述 | [doc/design/UI/](../../doc/design/UI/) | （按切片实例填）|
| 跨模块约束 | [arch/横切关注点.md](../arch/横切关注点.md) / 红线复核 | （按切片实例填）|

c. **真读**：不是"曾经读过"、不是"上下文里有的词"、不是"sticky 列表里有"。Owner verbatim：「在架构上工作，就得理解架构」。  
d. **退回纪律**：第①步及之后任一步发现新资料缺口 → **退回本步补料**，不凭印象续推。  
e. **AI 自做、不烦 Owner**（详 [守则 §4 专业自决与共创诚实](../守则.md)）：备料是 AI 本职，不列「这些要不要读请你拍板」。  
f. **v2 字面校验纪律**（**F1** / 第一七〇次 Stage 1 立 / 第一六六次「反甩锅样板第三号·v2 字面残词污染」教训沉淀）：凡参考来自 v2 文件（含 [Slice候选清单.md](../tp-业务切片/Slice候选清单.md) 中 v2 推倒过头的 B-* 骨架项 / [bang-v2/](../../../bang-v2/) 任意 spec / decisions / DEVNOTES / 既有 TP 笔记的早期残段等）的字面表述，**不得作为业务事实直接引用**；必须**反查 doc/domain + Owner verbatim + 现行 arch** 三方对齐后方可入 Slice。典型场景：v2 候选清单字面「顶部滑动切换其他用户」「切换发布者」「推荐用户」等被 Owner verbatim 否定（详 第一六六次 主-4 11 段 verbatim）。处置：若 v2 字面与现行权威冲突 → 以现行权威为准 + 在候选清单 / 笔记池标注「v2 字面已弃」。

**红线**：凭印象推理 = 立即停 + 退回本步。详 [守则 §3 真知而非印象](../守则.md)。

### 第 ① 步：定主线范围 + 节点拆分

- AI 提议主线骨架（节点序列、主线边界、不覆盖项）
- Owner 确认；范围模糊时此步停下来澄清，不带模糊范围进入下一步

### 第 ② 步：原则集梳理（双向 + 双源 + 冲突前置）

a. **AI 提取候选原则**：按权威序反查 `doc/domain/` → `plan/arch/` → `plan/DEVNOTES.md` → `doc/design/UI/`  
b. **Owner 校验业务方向 + AI 自决原则组织（D-129 改义）**：原则的提取、归并、措辞由 AI 用专业方法论（DDD / 信息架构 等）自决并 brief 报告；Owner 校验范围 = 业务方向是否漂、与 domain 是否冲突、是否漏了 Owner 已声明过的关键约束。**AI 不列「原则 ①②③ 你选哪条」让 Owner 替 AI 思考**。  
c. **反查结果分类**（A 辅来源同时确定）：

| 反查结果 | 处置 | A 辅来源 |
|---|---|---|
| 与 domain 一致 | 直接采用 | **来源 1**：文档权威落点（文件 + 章节 + 行号） |
| domain 无声明 | 标"新原则" | **来源 2**：Owner sign-off（日期 + 会话号 + verbatim） |
| 与 domain 冲突 | **【前置停】**列报告 → 双向修正二选一：① 原则描述错 → 修原则；② domain 表述错 → Owner 仲裁后修 domain（修 domain 需慎重，留 DEVNOTES 痕） | 仲裁后归来源 1 或 2 |
| 与既有 Slice 已有原则一致 | **import 引用，不重抄** | **来源 3**：旧 Slice 锚点 |

d. **原则可复用**：旧 Slice 已确立的原则原文不再重抄，按 import 形式引用，本 Slice 仅声明"适用 P-X（来自主-N §1）"  
e. **确认终版原则集**：进入下一步前 Owner 与 AI 共同确认本切片的原则集已封闭

### 第 ③ 步：形态选择（α / β / 二者并用）

- 在原则集确定后明确：本 Slice 用 α、用 β、还是 α+β 并用
- 判据：见 §2.γ 选择指引
- 并用情形：当一个切片既有"主线骨架可矩阵化的部分"又有"事件叙事自然的部分"时，两形态各管一段，文件内分节呈现

### 第 ④ 步：维度声明（§2.5 / 强制段 / Owner 审）

- AI 由原则集**推导**本切片涉及的维度集合（不机械列全清单，由原则集决定哪些维度参与）：
  - **骨架维度**（节点划分）
  - **变体维度**（列分叉）
  - **情境维度**（行枚举依据子维度：数据状态 / 业务参数 / 用户身份 / 行为历史 / 环境条件）
  - **行为维度**（格内必填项：可达性 / 行为路径 / 拒绝形态 / 数据写入 / 副作用 / 承载层 / 提示文案 / 路由参数 / 路由名 CF 名）
  - **显式声明"不涉及"项**（哪些子维度本切片不涉及 + 理由）
- Owner 一次拍板审：是否遗漏维度 / 是否过度展开

### 第 ⑤ 步：「原则 × 节点 × 维度变体」笛卡尔积落细则表

- 按声明的维度集机械组合，逐格填写
- 每条结论**强制反向引用 A 辅**（文档落点 或 Owner sign-off 或 import 自旧 Slice）
- 无 A 辅 → 自动标 ⚠️【G-N】缺口

### 第 ⑥ 步：§4 信息缺口集中一次拍板

- 仅"信息缺口"类（边角细节没人定过 / 文档暂未触及）
- "原则冲突"已在第 ②c 步前置处理过，不再混入

### 第 ⑦ 步：收尾回写四方向（**F2** / 第一七〇次 Stage 1 立 / 第一六九次教训①「反向链接回写漏洞」沉淀）

## § 4 三大配套机制

### 4.1 UI 补强子 TP（决议 4）
- **触发**：Slice § 1 标 ⚠️ 缺页面/缺要素
- **形式**：独立 TP（tp-UI-补强 / 可累积批处理 / 不阻塞 Slice 主线 1-2 篇）
- **产物**：doc/design/UI/[页面].md 结构化描述（页面要素 + 跳转 + 可见性条件 / 非视觉稿）

### 4.2 开发者 AI 自检 F1/F2/F3（决议 3）
- **位置**：入 [roles/开发者-instructions.md](../roles/开发者-instructions.md) 加段（Slice 进入开发者视野时强制走 F1/F2/F3 自检）
- **价值**：开发者要自己理解业务 / 给参谋长 AI 一个外部检查器 / Owner 工作量集中在仲裁不分散到验收

### 4.3 与任务书的正交关系（误解修正）
- **Slice ≠ 任务书的业务描述部分**
- **Slice 是横向穿透 N 个专题/模块**（业务流维度）
- **任务书是纵向聚焦 1 个专题/模块**（技术实现维度）
- **开发者使用模式**：接手主题预热读 Slice 集（大局观）/ 实现具体 BC 时读任务书（聚焦不冗余）/ 跨模块疑问时回查 Slice（按需）
- **任务书末尾**只列 Slice 链接索引，**不嵌入 Slice 内容**

## § 6 方法论根基（DDD / 软件工程）

本方法论汲取的现成实践：

| 实践 | 启发点 |
|---|---|
| **Specification by Example**（Gojko Adzic）| 例子比规则更难误读 / 啰嗦是机制不是缺点 |
| **Domain Storytelling**（Hofer & Schwentner）| 业务专家口述具体场景 / AI 即时落 Markdown / 强制统一语言 |
| **Use Case 2.0 Slice**（Ivar Jacobson）| 用例不是一次写完 / Slice 增量精炼 / 可暂停可恢复 |
| **BDD Given-When-Then** | 三段式强制省略号无处藏身 |

不堆名词。**Slice 是这些实践在 v3 项目场景下的适配混合**，核心引擎是 **Owner 怀疑驱动机制**（不是 AI 主动构造曲解）。

## § 7 关键纪律（必读）

1. **Owner 是检测器不是生成器**：方法设计便利 Owner 提问 / 不便利 AI 输出
2. **AI 诚实暴露犹豫**：§ 4 强制段 / 不写"圆满"的 Slice
3. **不改 domain**：新词汇入 doc/slices/术语表.md / domain 仅在发现表述错误时修订
4. **不预设篇数**：靠收敛信号停 / 不靠拍脑袋数
5. **三件事齐活**：A/B/C 任何一件缺 = Slice 不合格
6. **正交于任务书**：不嵌入 / 只索引
7. **domain 是原则源 / AI 是设计师+领域分析师 / Owner 是检测器+仲裁者+补充者**（D-126）：AI 不从 domain 反推原则后直接使用，必须反查校验；AI 不是任何一方的传声筒；Owner 不是唯一权威，最高权威是 domain
8. **原则可跨 Slice 复用**（D-126）：旧 Slice 已确立的原则原文不重抄，本 Slice 仅声明「适用 P-X（来自主-N §1）」；但每篇 Slice 必须单独确认引用与适用范围
