# 需求收集协议

> 版本: v1.0 / 来源: csf-v4-live / 纳入: coslog-011
> 性质：操作规程。非阶段限定——需求收集贯穿项目全程，与方案设计、开发交织进行。
> 上游：守则.md（元原则）、协作规范.md（角色通信）

---

## §1 协议定位

本协议定义 CSF 项目中的需求收集方式。

**需求收集不是阶段**——它不与"方案设计"或"开发"形成先后关系。需求在散点讨论中浮现，在方案对比中澄清，在开发中继续发现。本协议定义的是**无论处于什么阶段，当需求出现时如何捕获**。

**需求来源**（非单一）：
- Owner 直接描述（「我需要XXX」）
- Owner 提出技术设想/场景方案（反向提取）
- 项目目标/标题本身隐含的宏观需求
- 方案对比过程中暴露的约束和取舍
- 开发过程中发现的遗漏

---

## §2 参谋长的双重职责

### 2.1 需求提取

从 Owner 的任何输入中识别需求信号，翻译为业务语言的需求陈述。

### 2.2 方案建议

在理解需求方向后，**主动调用专业知识**提出可行的技术方案。不止一个——给出对比，说明各方案的适用条件、代价和边界。Owner 在对比中选择或提出新方向。

**禁止两种极端**：
- 只提取需求不提供方案（等于把专业判断推给 Owner）
- 需求未确认就跳到架构设计（等于在浮沙上盖楼）

**正确节奏**：需求信号→确认方向→方案对比→Owner选择→细化→归入需求池。

### 2.3 确认边界

何时必须确认，何时直接执行：

| 操作类型 | 确认规则 |
|---|---|
| 需求池写入（收集类） | 参谋长自行写入，不逐条确认。Owner 在场会纠偏 |
| 正式文档/代码 | 动手前必须 Owner 确认 |
| 需求方向不明、多重解读可能 | 提请 Owner 澄清后再写 |
| 方案设计写入 DEVNOTES | 方向性决策需确认，技术细节自行写入 |

## §3 需求书写规范

### 3.1 核心原则：用场景和故事，不用功能点

需求的主体不是"系统应提供XXX功能"，而是**谁在什么情况下遇到什么问题、为什么需要解决**。

### 3.2 需求条目格式

每条需求包含：

| 字段 | 说明 |
|---|---|
| **编号** | R-xxx，按录入顺序 |
| **场景** | 一个具体的故事：谁，在什么情境下，遇到了什么问题。让读者能「看到」问题。 |
| **需求** | 从场景中提炼的业务需求陈述（一两句）。 |
| **归域** | 归入项目定义的需求域分类 |
| **来源** | session-NNN-回合N / Owner直接描述 / 项目目标推导 |
| **状态** | ✅已确认 / 🟡待确认 / ❌废弃 / 🔀已合并 |
| **关联方案** | 若有对应的技术方案/架构设计，在此标注路径（可选） |

反面示例（功能点语言，禁止）：
> 「系统应提供场景化的模板，支持参数化和分组。」

正面示例（场景+故事语言）：
> **场景**：张经理刚和AI讨论完一个立项方案。他需要让AI收尾，但打开对话窗口后，面对空白输入框，一时想不起标准措辞。他切出去找文件、翻文档、回来输入——这个过程打断了他的思考流。
>
> **需求**：用户在需要执行标准操作时，不应被"该怎么说"打断工作节奏。系统应在输入环节让用户快速找到并调用对应场景的标准措辞。

### 3.3 场景的粒度

一个场景 = 一个完整的用户故事片段，包含情境、动机、痛点。不要太细（变成交互说明），不要太粗（变成需求域标题）。

---

## §4 需求池管理

### 4.1 持续收集，不等收尾

每次会话中产生需求信号 → 立即写入需求池。不堆积到收尾。

### 4.2 合并与冲突

- **同一需求的细化** → 合并到已有条目，追加场景细节
- **同一需求的不同表述** → 保留更清晰的那条，旧条目标「🔀已合并至R-xxx」
- **矛盾** → 两条都保留，标「⚠️与R-xxx矛盾」，提请 Owner 裁决

### 4.3 方案与需求的关系

需求池可以关联方案文档（在「关联方案」字段标注路径），但需求本身不描述方案。方案归入 DEVNOTES 或独立方案文档。

### 4.4 优先级不在此定义

需求池只负责收集和澄清。优先级、分期在立项阶段确定。

---

## §5 不因"在需求探索"而停止方案思考

需求收集与方案设计交织进行。当 Owner 提出设想时，参谋长的完整响应是：

1. 理解方向 → 确认需求信号
2. 提供方案对比（主动）
3. Owner 选择/纠偏
4. 需求归入需求池 + 方案要点记入 DEVNOTES

不因"还在收需求"而自我设限，拒绝思考方案。也不因"想到了好方案"而跳过需求确认。
