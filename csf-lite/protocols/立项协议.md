# 项目立项与任务窗口

> 本文件覆盖两件事：**立项**（为将要做的事建结构）和**任务窗口**（任务阶段的切换机制）。
> 两者放在一起，因为立项有时会触发窗口切换，两件事需要衔接理解。

---

## § 0 任务窗口是什么

**任务窗口 = cos-context.md §B 全景图中的当前焦点区域。** 三元组（目的/方法/资源）存放在独立的 `triplets/` 文件中，§B 通过指针引用活跃三元组。

在一个窗口内：
- 目的统一，方法稳定可复用，资源边界固定——这些由三元组文件承载
- 多个会话共享同一个三元组文件——变的只有 §C/§D 和红点位置
- 三元组文件不动，红点动，是正常状态

**窗口 vs 三元组**：窗口（§B）提供全景导航（全景图 + 三元组指针 + 通用资源），三元组文件提供执行内容（目的/方法/资源/进度/备忘）。两者解耦：切换三元组不触发窗口切换，只需修改 §B 指针。

**滑入 / 滑出**：
- 滑入 = 当前三元组文件的方法和资源能覆盖接下来一段时间的工作
- 滑出 = 需要装载不同的三元组文件，即将进入不同工作模式

## § 1 何时立项 / 何时不立项

不是所有事都需要立项。判断标准：

| 维度 | 立项 | 不立项（用 todolist 追踪即可）|
|---|---|---|
| 复杂度 | 跨多个会话、需分步推进 | 一两个会话内能做完 |
| 业务意义 | 有独立业务目的、值得未来回溯 | 辅助性、工具性、一次性 |
| 重要性 | 失败/遗漏有显著后果 | 做不做影响有限 |
| 可追踪性 | 需要在全景图中有位置 | 完成即遗忘 |

**灰区决断**：如果参谋长不确定，向 Owner 提议两个选项（立项 vs todolist），说明理由，让 Owner 决断。

## § 2 立项触发

| 触发方式 | 典型场景 |
|---|---|
| Owner 指令 | "建立一个 TP 来做 X" / "起个计划" / "我们制定一个新的计划" |
| 参谋长提议 | 发现缺口 / 评估复杂度后建议立项，经 Owner 确认 |
| 流程驱动 | FLDD 按队列取下一个主题包进入 TP 周期 |

## § 3 立项动作序列

无论何种计划（SP / TP / 子 TP / ad-hoc），立项 = 依次完成以下步骤：

### Step 1：建文件夹 + README

**文件夹命名**：`{类型}-{NNN}-{主题简述}/`

- **NNN** = 立项时的会话编号（不是预计开始执行的会话编号）
- **类型**：sp / tp / ad-hoc
- **位置**：建在父任务文件夹内（如 sp5 下的 TP 建在 sp5 文件夹内）

**README 模板**（最小必备结构）：

```markdown
# {类型}-{NNN} {主题名}

> 创建：session-NNN / {日期}
> 性质：{功能开发 / 架构设计 / 打岔修复 / 漫游探索 / ...}
> 来源：{为什么立项——一句话}
> 优先级：{P0/P1/P2/— }

---

## § 1 目的

{做什么 + 为什么重要。完成后世界有什么不同。}

## § 2 分解

{预计分几步完成（IT / sub-TP / milestone），每步一句话描述业务能力。
  允许粗略——随推进细化。但立项时至少给出骨架。}

## § 3 资源

{精简列出：完成此计划需要访问/精读的文件路径。
  原则：尽量少且必要。不需此刻访问，但需明确知道在哪。}
```

> 额外章节（设计决策、活跃防御、依赖图等）视复杂度按需添加，不强制。

### Step 2：更新全景图

在 context.md §B 全景图中**插入新节点**：

- 在适当的层级位置（SP 下 / TP 下 / 独立分支）
- 展示预计的子步骤/子计划（前瞻视角）
- 标注当前状态（⬜ 设计 / 立项）

**全景图的作用 = 前瞻**：读者看到全景图时应能理解"接下来大致要走哪几步"。不必精确，但要有骨架。

### Step 3：三元组归入判断

判断新 TP 能否归入已有三元组文件：

| 判断 | 操作 |
|---|---|
| 新 TP 的工作类型与某已有三元组一致 | 在该三元组文件的进度表中追加一行 |
| 新 TP 需要不同的目的/方法/资源组合 | **新建三元组文件**（`triplets/` 下，参考 `_index.md` 格式）+ 更新 `triplets/_index.md` |
| 不确定 | 默认归入最接近的已有三元组。执行中发现不适配再拆分 |

> **原则**：三元组按工作类型（目标类型）划分，不按 TP 1:1 划分。一个三元组可承载多个 TP，一个 TP 也可使用多个三元组（TP 分解出的细分任务若涉及不同工作类型，可分别归入对应的三元组）。新 TP ≠ 新三元组。

### Step 4：更新进度表

在对应三元组文件的进度表中追加一行（新 TP 及其预计 IT 分解）。

### Step 5：移动红点

参谋红点是否移到新计划 = 视优先级判断：
- 新计划是 P0 / 插队 → 红点移过来
- 新计划是排队 → 红点不动，仅注册

### Step 6：更新开发者进度表

如果新计划涉及开发工作，在 dev-context.md §B 进度表中插入对应行（待出 STB 状态）。

### Step 7：判断是否需要切换三元组

**判断标准**：新计划使用的是当前活跃三元组文件，还是不同的三元组文件？

| 情况 | 处理 |
|---|---|
| 新计划归入了当前活跃三元组 | 不切换。更新 §B 三元组指针无需改动 |
| 新计划使用了不同的三元组文件 | 在 §D 中标注下次会话使用的三元组文件；下次开局回合 1 时切换指针（见 §4）|
| 不确定 | 默认保持当前三元组。如果执行中发现不适配，再切换 |

### Step 8：Owner 校验

向 Owner brief 立项结果：
- 一句话说目的
- 列出分解骨架
- 说明归入的三元组文件（新建立 or 已有）
- 说明优先级和红点位置
- 等 Owner 确认或纠偏

## § 4 三元组切换规程

### 触发条件

Owner 宣布切换方向 / 参谋长报告"当前三元组内计划事项已全部完成"后 Owner 决断。

**AI 不自行判断三元组边界**：话题游移是正常的，Owner 在掌舵。只有 Owner 明确决定方向切换后，才执行本规程。

**关键纪律**：三元组切换不能用普通收尾（只覆写 §C/§D）来代替。切换是独立操作，往往值得单独一个会话完成。

### Leave（离开当前三元组）

1. **更新当前三元组文件**：进度表 + 备忘（即正常收尾动作，见 cos-context.md §A 收尾协议）
2. 确认会话日志已归档
3. 三元组文件本身不删除、不封存——保留为完整回溯链

### Enter（进入新三元组）

4. **确认三元组文件已存在**：如果 §3 Step 3 已新建，直接使用；否则先用 §3 Step 3 建文件
5. **主动要求 Owner 补充/确认**新三元组的目的和资源（尤其是资源——资源定义了 AI 的视野边界，不明确就是盲区）
6. 更新 `cos-context.md §B` 三元组指针 → 指向新三元组文件
7. 填写 §D（新三元组下第一个会话的计划）

### Return（回归之前的三元组）

当新工作完成或暂停、需要回到之前的三元组时：
1. 更新 `cos-context.md §B` 三元组指针 → 指回之前的三元组文件
2. 精读该三元组文件，检查进度表和备忘是否仍然准确（期间可能有变化）
3. 恢复红点到合适位置
4. 填写 §D

> **三元组文件即快照**：旧模型需要在父 README 中备份"窗口快照"以防丢失；新模型下三元组文件始终独立存在，进度表持续累积——切换只是改指针，不丢任何信息。

## § 5 全景图与红点

**全景图** = 项目整体发展脉络的参谋长认知载体。是 L2 级的「我在哪、从哪来、到哪去」。

包含 S 级和 T 级细分，必要时记录到三四五级。最小细分单元 = 一个会话。

运作规则：
- 历史结构不可变（已发生的节点锁定）
- 未来节点可增删修改
- 表达粒度可调（精简/展开）——全景图是活的

### 5.1 两种表达模式

全景图有两种表达模式，由项目的实际工作状态决定。一个项目可以混合使用——已完成的历史保留为阶段树，当前活跃区用漫游模式。两者不是非此即彼：一个 GP 下可以先有一段阶段模式，再进入漫游模式，之后再切回阶段模式。

| 模式 | 适用场景 | 结构特征 |
|---|---|---|
| **阶段模式** | 三元组分段变化，工作按顺序推进（如 S1→S2→S3） | 树状 S→TP→IT，节点互斥、有先后依赖 |
| **漫游模式** | 三元组文件稳定、多线索并行、主题游离 | 线索图，线索并行存活、独立推进；不同线索可能共享同一三元组文件 |

**模式标注**：全景图顶部用 HTML 注释标注当前活跃模式：
- `<!-- @MODE:阶段 -->` — 当前按阶段推进，红点打在 S/TP/IT 节点
- `<!-- @MODE:漫游 -->` — 当前多线索并行，红点打在活跃线索

### 5.2 模式切换

| 切换方向 | 触发条件 | 操作 |
|---|---|---|
| 阶段→漫游 | 当前阶段闭合后，后续工作不再适合用阶段表达 | Owner 确认 → 全景图追加漫游线索区，模式标更换 |
| 漫游→阶段 | 出现明确的分段三元组变化（如"下一个大版本启动"） | Owner 确认 → 从漫游中切出，建新阶段节点 |

**纪律**：切换时原结构不删除——阶段树作为历史保留，漫游线索区追加在下方。禁止把线索伪装成阶段插入阶段树。

### 5.3 线索（Thread）

线索 = 漫游模式下一条长期存活的并行工作流。按活动性质分类，不由时间顺序定义。

| 类型 | 图标 | 含义 | 生命周期 |
|---|---|---|---|
| 建设 | 🏗️ | 有明确终点的构建任务（≈ 传统 TP） | 完成即关闭 ✅ |
| 改进 | 🔧 | 持续的打磨、重构、体验优化 | 长期活跃，可暂停 💤 |
| 修复 | 🐛 | 缺陷响应 | 按需激活/休眠 |
| 发布 | 📦 | 版本构建与交付 | 周期性触发 |
| 探索 | 🧭 | 方向不明确的研究/试错 | 随时切出转为正式 TP |

**线索操作**：
- **新增**：参谋长提议，Owner 确认。在全景图中加一行（比 TP 立项更轻）
- **关闭**：标 ✅，不删除（历史不可变）
- **暂停**：标 💤，保留结构；下次激活时去掉 💤
- **TP 归入**：立项时在 README 中标注所属线索；漫游模式下全景图的 TP 节点缩进在对应线索下

### 5.4 红点

**红点** = 当前进展位置标记（参谋红点 / 开发红点 独立标记）。

**阶段模式下的红点**：
- 红点打在当前的 S/TP/IT 节点
- 不限于窗口切换时移动——每个会话推动进度变化时，都检查是否需要移动红点
- 窗口不动，红点动，是正常状态

**漫游模式下的红点**：
- 每条活跃线索有独立红点，标记该线索内的当前进展
- GP 级标记「当前焦点线索」（Owner 切换注意力时移动）
- 线索间切换不触发三元组切换（同属一个三元组文件覆盖范围）

### 5.5 全景图结构纪律

1. **禁止假阶段**：如果一个"阶段"的进入/退出条件无法用一句话说清，它不是阶段——是线索。禁止把线索当阶段插入阶段树
2. **先有工作再有节点**：不为"将来可能会做"的事预先占位。线索可以预先声明（因为它是维度不是任务），但线索下的 TP 只在立项时才插入
3. **历史不可变**：已闭合的节点（✅）锁定，不删除不移动
4. **模式标注必填**：全景图顶部必须用 `<!-- @MODE:阶段/漫游 -->` 标注当前模式，未标注 = 协议违规

## § 6 特殊类型：漫游计划（Ranger Task）

有时进入一种没有特定任务、东拉西扯的状态——探索、讨论、调研、或 Owner 想随意聊。这也可以立项：

**命名**：`ad-hoc-NNN-漫游/` 或直接在会话中标注"漫游模式"

**特征**：
- 无预设目的（或目的 = "看看有什么值得做"）
- 无预设分解
- 允许随时切出（发现值得做的事 → 就地立项或记入备忘）

**最小要求**：
- README 可以只有一句：`性质：漫游探索。无预设目的。`
- 全景图不必插入（漫游不占正式位置）
- 会话笔记照做（回合记录不省）

**退出**：漫游中产生了明确方向 → Owner 决断是否转为正式 TP。

> **与线索的关系**：漫游计划运行在全景图 🧭探索 线索下。当漫游产生明确方向并转为正式 TP 时，该 TP 归入对应的 🏗️建设 或 🔧改进 线索。详见 §5.3。

## § 7 立项完成判定

立项 = 以下全部完成：

- [ ] README 存在且 §1（目的）+ §2（分解）非空
- [ ] 全景图中有对应节点（漫游计划除外；漫游模式下 TP 须标注所属线索）
- [ ] 三元组归入已完成（归入已有 or 新建三元组文件）
- [ ] 对应三元组文件的进度表已追加行
- [ ] 红点位置已检查/更新（漫游模式下须更新对应线索的红点）
- [ ] 开发者进度表有对应行（如涉及开发）
- [ ] 三元组切换已处理（如需要，见 §4）
- [ ] Owner 已校验
